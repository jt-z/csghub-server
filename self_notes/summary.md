这是一个基于 Go 语言的微服务架构项目，名为 `csghub-server`。以下是根目录的组织结构说明：

## 项目结构概览

### 服务层目录

| 目录 | 说明 |
|------|------|
| `api/` | API 服务，处理 HTTP 请求和响应，是外部客户端的入口点 |
| `user/` | 用户服务，处理用户注册、登录、个人资料管理等操作 |
| `accounting/` | 计费服务，处理用户 token/硬件资源使用记录、余额更新 |
| `moderation/` | 内容审核服务，处理文本和图片的违规内容检测 |
| `dataviewer/` | 数据预览服务，处理数据集元数据获取和文件预览 |
| `notification/` | 通知服务，处理邮件、推送等通知发送 |
| `aigateway/` | AI 网关服务，处理 AI 模型推理操作，是 AI 服务的入口 |
| `runner/` | Runner 服务，作为 API 服务与 Kubernetes 集群之间的桥梁 |
| `logcollector/` | 日志收集服务，收集 Kubernetes 集群日志 |

### 核心组件目录

| 目录 | 说明 |
|------|------|
| `builder/` | 构建层，处理数据库访问、RPC 调用、Git 操作等底层操作 |
| `component/` | 组件层，处理业务逻辑，协调各层之间的交互 |
| `common/` | 公共工具库，包含通用类型定义、工具函数等 |
| `configs/` | 配置文件目录，包含推理、评估、空间等配置 |

### 基础设施目录

| 目录 | 说明 |
|------|------|
| `cmd/` | 命令行入口，包含主程序和各种子命令（如 cron、mirror、trigger 等） |
| `docker/` | Docker 相关配置，包含空间构建的 Dockerfile 和模板 |
| `mq/` | 消息队列相关代码 |
| `multisync/` | 多源同步相关服务 |
| `mirror/` | 镜像仓库同步服务 |
| `scripts/` | 辅助脚本文件 |

### 测试与工具目录

| 目录 | 说明 |
|------|------|
| `_mocks/` | 自动生成的 Mock 文件（由 mockery 生成） |
| `linters/` | 代码检查器配置 |

### 根级别文件

| 文件 | 说明 |
|------|------|
| `go.mod` / `go.sum` | Go 模块依赖管理 |
| `Makefile` | 构建、测试、lint 等命令脚本 |
| `docker-compose.yml` | Docker Compose 配置 |
| `.gitlab-ci.yml` | GitLab CI/CD 配置 |
| `AGENTS.md` | 代理服务说明文档 |
| `README*.md` | 项目说明文档（多语言版本） |

## 架构设计特点

该项目采用**分层架构设计**：
- **Handler 层**：处理 HTTP 请求和响应
- **Component 层**：处理业务逻辑，协调不同层之间的交互
- **Builder 层**：处理底层操作（数据库、RPC、Git 等）

每个服务都遵循这个分层模式，确保代码结构清晰、职责分明。

## 架构说明

### 三层架构模式

| 层级 | 职责 | 对应目录 |
|------|------|----------|
| **Handler Layer** | 处理 HTTP/gRPC 请求，参数校验，路由分发 | `api/`, `user/`, `accounting/`, `moderation/`, `dataviewer/`, `aigateway/`, `runner/`, `notification/`, `logcollector/`, `mirror/` |
| **Component Layer** | 业务逻辑处理，协调各模块交互 | `component/`, `common/types/` |
| **Builder Layer** | 底层资源操作封装（数据库、Git、部署等） | `builder/store/database/`, `builder/deploy/`, `builder/git/` |

### 服务间调用关系

```
外部请求
    │
    ▼
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌────────────┐
│   API    │────▶│ Component │────▶│  Builder │────▶│  Database  │
│ Handler  │     │   Layer  │     │   Layer  │     │   / Git    │
└──────────┘     └──────────┘     └──────────┘     └────────────┘
    │
    │ 代理请求到其他服务
    ▼
┌──────────┐     ┌──────────┐     ┌──────────┐
│  User    │     │Accounting│     │Moderation│
│ Service  │     │ Service  │     │  Service │
└──────────┘     └──────────┘     └──────────┘
```

### 关键数据流

1. **API 入口** → Handler 处理 → Component 业务逻辑 → Builder 数据操作
2. **AIGateway** 独立处理 AI 推理请求，直接与模型服务交互
3. **Runner** 作为 API 与 Kubernetes 之间的桥梁，处理部署任务
4. **Notification** 通过 Temporal Workflow 异步处理通知发送

这个架构设计实现了**关注点分离**，使得各层职责清晰，易于维护和扩展。
